Modelo de Base de Datos NoSQL (Firestore)

El proyecto "iBichos" utiliza Firebase Firestore, una base de datos NoSQL. A diferencia de las bases de datos tradicionales (como MySQL) que usan tablas relacionadas entre sí, Firestore guarda la información en un formato de Colecciones y Documentos.

1. ¿Por qué NO hay una colección para el "Ranking"?

En nuestra base de datos no existe una colección separada para el Ranking. Esto no es un error, sino una decisión técnica correcta para evitar problemas estructurales. Las razones principales son:

  • Evitar Datos Duplicados: Los puntos de experiencia (XP) le pertenecen al jugador y ya están guardados explícitamente en su perfil (users). Si creáramos una colección extra solo para el ranking, tendríamos el mismo dato guardado en dos lugares distintos. Esto nos obligaría a sumar los puntos en dos partes a la vez cada vez que se atrapa un bicho. Si algo falla a nivel de código o de internet, los datos quedarían descuadrados (alguien podría subir de nivel pero no subir puestos en el ranking). En desarrollo de software, lo correcto es tener los datos en un solo esquema (Una única fuente de verdad).
  • Firestore es muy veloz ordenando: Nuestra base de datos está diseñada para buscar y ordenar datos al instante mediante índices. Para armar el Ranking, el sistema simplemente consulta la colección principal de jugadores (users) y solicita: "Entrégame a los mejores usuarios ordenados de mayor a menor según su variable XP".

2. Estructura de la Base de Datos

A continuación se detalla cómo se guarda la información. Se compone de dos colecciones principales en la base de datos.

Colección: users

Guarda el progreso de los jugadores. Cada documento representa a una persona registrada y su identificador corresponde directamente al código privado que le asigna Firebase Authentication.

  • ID del Documento: [Código único de Auth] (Ej: AB12cdEF34...)
    • displayName (Texto): El nombre que eligió el usuario (Ej: "Juan Pérez").
    • email (Texto): Su correo de ingreso.
    • xp (Número): El total de puntos que ha acumulado. Es la variable clave para calcular su rango y para definir su posición en la pantalla de Ranking.
    • level (Texto): El título de su rango actual (Ej: "Casual", "Aprendiz", "Maestro").
    • avatarUrl (Texto): El link de internet donde está guardada su foto de perfil.

Colección: captures

Guarda todas las atrapadas de la comunidad de la aplicación.

  • ID del Documento: [Generado al azar por Firebase] (Ej: Xk8sP9Lz...)
    • userId (Texto): Conecta la captura con el jugador. Corresponde al código único de quien lo atrapó en la colección de usuarios.
    • imageUrl (Texto): El link directo a la foto que se le tomó al insecto.
    • insectName (Texto): Nombre común del insecto asertado por la Inteligencia Artificial.
    • scientificName (Texto): Nombre científico de especie provisto por la IA.
    • probability (Número): Porcentaje de confianza de la IA en la detección (Ej: 0.98 = 98%).
    • dangerLevel (Texto): Clasificación del nivel de peligro (Ej: "Inofensivo", "Peligroso").
    • latitude / longitude (Número): Coordenadas del equipo móvil donde ocurrió la captura.
    • capturedAt (Número): Fecha y hora estilo "Timestamp" para ordenar la pokédex personal por fechas.
    • xpAwarded (Número): Los puntos exactos que entregó este insecto en el instante de su validación.

3. Relaciones Intracolección (Consultas lógicas)

Al carecer de sentencias "JOIN" típicas de las bases de datos rígidias, la aplicación conecta las cosas lógicamente en tiempo real:

  • Para mostrar la Pokédex de alguien (Colección de Usuario):
    La aplicación solicita a captures todos aquellos componentes donde el atributo userId sea idéntico al perfil activo.
  • Para el Ranking General:
    La aplicación se encarga de solicitar los documentos iniciales a la colección users bajo un ordenamiento descendente de la variable xp y la limita, evitando traer datos de más.
