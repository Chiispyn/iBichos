💻 Arquitectura Técnica: Dashboard iBichos (React)

Este documento es la referencia técnica definitiva para el panel de control web. Explica la arquitectura de una SPA (Single Page Application), los patrones de seguridad para aplicaciones empresariales y la integración de gráficas analíticas interactivas.

1. Stack Tecnológico de Renderizado

Framework

React 19 + Vite

Elegimos React por su paradigma de componentes reutilizables. Vite se encarga del empaquetado (bundling) ofreciendo Hot Module Replacement en menos de 50ms, acelerando drásticamente el tiempo de desarrollo.

Lenguaje

TypeScript (TSX)

Provee tipado estático fuerte. Definimos interfaces (ej: interface Usuario { id: string... }) que aseguran que las respuestas de la base de datos coincidan exactamente con lo que el código espera renderizar.

Data Viz

Recharts

Una librería composable construida directamente sobre elementos SVG de React. Permite renderizar miles de nodos de datos en gráficos de pastel y barras interactivos sin afectar el rendimiento del DOM.

2. Patrón de Seguridad: Role-Based Access Control (RBAC)

A diferencia de la app móvil (abierta al público), el Dashboard es una intranet cerrada. Su seguridad se basa en el patrón de "Rutas Protegidas" acoplado al estado de Firebase Auth.

Arquitectura del Contexto (AuthContext):

En lugar de consultar Firebase en cada página, creamos un Provider global. Este archivo (authcontext.tsx) establece un onAuthStateChanged listener.

Doble Validación: Cuando alguien se loguea, no basta con que el token sea válido. El contexto hace una consulta asíncrona oculta: getDoc(doc(db, "admins", currentUser.uid)). Solo si este documento existe y el campo es estado: "activo", el contexto otorga la bandera isAdminActive = true.


// El Guardián de Rutas (protectedRoute.tsx)
export const ProtectedRoute = ({ children }: { children: JSX.Element }) => {
  // Consumimos el contexto global
  const { user, isAdminActive, loading } = useAuth();

  if (loading) return <p>Cargando sesión...</p>;

  // El núcleo de seguridad: Si no hay usuario, o el contexto dice que no es admin, lo expulsamos.
  if (!user || !isAdminActive) {
    return <Navigate to="/" replace />;
  }

  // Si sobrevive a la validación, renderizamos el panel.
  return children;
};

3. Lógica de Trazabilidad: Daily Active Usage (DAU)

En la pantalla de Analítica, convertimos datos en bruto en conocimiento de negocios (Business Intelligence). El caso más notable es la gráfica de Retención Diaria.

El Algoritmo de Agrupación:

  1. Descargamos miles de documentos en crudo de la colección sessions.
  2. En lugar de pintar puntos incomprensibles, iteramos sobre cada documento y extraemos el Timestamp de inicio (startedAt).
  3. Convertimos ese Timestamp a un String legible localizado usando toLocaleDateString('es-CL') (ej: "23 abr").
  4. Usamos un diccionario Hash Record<string, number> para sumarizar matemáticamente todos los minutos (durationMinutes) que comparten la misma llave (la fecha).
  5. Finalmente, el diccionario es transformado en un Array y arrojado al componente <LineChart> para crear una curva histórica de retención (Screen Time agregado).

4. Lógica de Negocios de Moderación (Galería End-to-End)

La pantalla capturas.tsx es el cierre del ciclo End-to-End. Lo que un usuario en Android captura en las calles, aparece aquí instantáneamente para ser juzgado por un humano.

El algoritmo de filtrado descarga todas las capturas (getDocs), pero la función sort() de JavaScript ha sido sobreescrita para que cualquier objeto cuyo campo sea status === 'PENDING' sea forzado a la posición del índice 0, garantizando que el moderador siempre vea el trabajo pendiente sin tener que buscar.

Los botones de "Aprobar" y "Rechazar" no solo cambian la interfaz gráfica. Detonan una operación asíncrona de mutación updateDoc sobre Firestore, cambiando el campo status a APPROVED y needsReview a false de forma permanente, lo cual posteriormente impacta en las estadísticas de la pantalla de Analítica.
