📱 Arquitectura Técnica: iBichos App (Android)

Este documento es la referencia técnica definitiva para el equipo de desarrollo. Detalla en profundidad las decisiones de diseño, patrones de código y flujos de datos utilizados en la aplicación móvil nativa de iBichos, desarrollada en Kotlin.

1. Patrón Arquitectónico: Clean Architecture + MVVM

El proyecto no utiliza la arquitectura tradicional monolítica de Android. Implementamos Clean Architecture para lograr el principio de "Separación de Preocupaciones" (Separation of Concerns). Esto aísla la lógica de negocio de los frameworks externos (como Firebase o la Interfaz Gráfica).

Las Tres Capas Estrictas:
  • Data Layer Es la única capa que interactúa con la red. Aquí residen UserRepositoryImpl y CaptureRepositoryImpl. Utiliza Corrutinas de Kotlin para manejar solicitudes asíncronas a Firestore sin bloquear el hilo principal (Main Thread).
  • Domain Layer Contiene entidades puras en Kotlin (ej. data class User, enums como UserLevel) y Repositorios como interfaces. Esta capa no tiene dependencias de Android. Es 100% testeable mediante Unit Testing aislado.
  • Presentation Layer Construida con Jetpack Compose. Las pantallas no tienen lógica condicional de base de datos; simplemente "observan" estados a través de un ViewModel usando StateFlow.

2. Flujo Crítico de Negocio: La Toma de Captura

La función central de la aplicación (Identificar Insectos) involucra un orquestamiento complejo de 3 APIs distintas que ocurre en menos de 3 segundos:

  1. Hardware (CameraX): El usuario abre la cámara. Se utiliza la librería CameraX de Android Jetpack para capturar un ImageProxy y convertirlo a un archivo temporal local.
  2. Almacenamiento (Cloudinary): Para evitar los cuellos de botella de facturación de Firebase Storage, la imagen se sube mediante una petición HTTP POST a la API REST de Cloudinary. Cloudinary procesa la imagen y devuelve una secure_url pública.
  3. Inteligencia Artificial (Kindwise API): El ViewModel toma esa URL pública y dispara un request a la API de Kindwise (https://api.plant.id/v2/identify u homóloga de insectos).

    Mapeo de Datos Kindwise: Kindwise devuelve un JSON complejo. El repositorio extrae:
    • suggestions[0].plant_details.scientific_name
    • probability (Certeza de la predicción, double 0.0 - 1.0).
  4. Clasificación Interna: La app toma el nombre científico y ejecuta nuestra lógica de taxonomía, asignando un enum TaxonomicCategory (ej. ARACHNIDA) y calculando el dangerLevel (ej. VENOMOUS).
  5. Persistencia (Firestore): Se construye un objeto inmutable Capture y se inserta como documento en la colección captures asociado al UID del usuario mediante el campo userId.

3. Motor Centralizado de Gamificación

Para evitar que el código sea ilegible, la experiencia del usuario (XP) no se calcula en las pantallas, sino en un motor centralizado estático o inyectado llamado GamificationConfig.


// Ejemplo conceptual del motor de Gamificación
object GamificationConfig {
    // Escala de Puntos Asimétrica
    fun calculateXp(category: TaxonomicCategory): Int {
        return when(category) {
            TaxonomicCategory.ARACHNIDA, TaxonomicCategory.HYMENOPTERA -> 300
            TaxonomicCategory.COLEOPTERA -> 200
            else -> 100
        }
    }

    // Máquina de Estados para Ligas
    fun calculateLevel(totalXp: Int): UserLevel {
        return when {
            totalXp >= 10000 -> UserLevel.BUG_MASTER
            totalXp >= 5000 -> UserLevel.ENTOMOLOGIST
            totalXp >= 2500 -> UserLevel.EXPLORER
            totalXp >= 1000 -> UserLevel.AMATEUR
            else -> UserLevel.CASUAL
        }
    }
}

4. Trazabilidad de Sesiones y Analítica (Screen Time)

El sistema recopila datos de manera pasiva para nutrir de información al Dashboard Web. Esto se logra mediante la interceptación del ciclo de vida (Lifecycle) de Android cruzado con la sesión de autenticación.

  • En MainActivity, se declara un FirebaseAuth.AuthStateListener.
  • Cuando un usuario válido se detecta, se genera un documento en la colección sessions con un Timestamp startedAt.
  • Cuando el usuario minimiza la app (OnStop) o cierra sesión, el sistema calcula la diferencia en milisegundos con la hora actual, y guarda el valor durationMinutes. Esto nos permite generar métricas de retención diarias exactas.

5. Gestión de Errores e Integridad

Toda comunicación hacia Firebase se realiza dentro de bloques try-catch y retornando a la UI un envoltorio (Wrapper) de estado, comúnmente llamado Result<T> (o Resource). Esto garantiza que la interfaz de usuario de Jetpack Compose siempre pueda mostrar un estado de "Cargando", "Éxito" o un "Error amigable" si no hay conexión a internet, evitando que la aplicación sufra un Crash fatal.
